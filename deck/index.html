Background
What is Paperclip?
  • 🏢
    Org chart for AI agents — not an agent framework. It manages the company that runs on top of agents.
    github.com/paperclipai/paperclip · 55,600★ MIT · launched Mar 4, 2026
  • 🤖
    Bring your own agents — OpenClaw, Claude, Codex, Cursor, any HTTP endpoint. "If it can receive a heartbeat, it's hired."
    Agents have roles (CEO, CTO, engineer, researcher), report lines, and monthly budgets
  • 💰
    Hard budget controls — monthly per-agent token budget. When it hits the limit, the agent stops. No runaway spend.
    Atomic task checkout prevents double-work across concurrent agents
  • 📋
    Tickets + goal ancestry — every task traces back to the company mission. Agents always know the "why."
    Immutable audit log, heartbeat scheduling, full tool-call tracing
Use Case
The monday.com Fit
Hypothesis

"When you have 10+ AI agents running in parallel — coding, researching, reviewing — Paperclip is the management layer that keeps them coordinated, budgeted, and auditable."

  • ✅
    20 Claude Code tabs open? Paperclip turns that into a ticket queue with assignments, status, and context threading.
  • ✅
    OpenClaw as a Paperclip employee — explicitly supported. OpenClaw is listed as a first-class adapter. "OpenClaw is an employee; Paperclip is the company."
  • ✅
    Directly relevant for AI Innovation team — org-wide agent config, reproducible setups, agent governance.
  • ✅
    On-prem / self-hosted — single Node.js process, no Paperclip account required, full data isolation.
Methodology
What We Actually Ran
  • ①
    Install + onboard — npx paperclipai onboard --yes. Resolved Docker Postgres workaround (root user incompatibility with embedded PG).
  • ②
    REST API exploration — created company, project, 3-agent org chart (CEO → Radar + CodeAgent), and 2 issues (MON-1, MON-2) via the API.
  • ③
    Heartbeat authentication — JWT generation, WS gateway connect, device pairing, scope grants. Diagnosed and fixed 3 separate auth blockers.
  • ④
    End-to-end heartbeat loop — Paperclip → process adapter → openclaw agent CLI → this session → HEARTBEAT_OK → Paperclip succeeded.
Architecture — Live POC
How the Loop Actually Works
Paperclip → Fires scheduled heartbeat on Radar agent (cron or on_demand)
→ Process adapter runs openclaw-heartbeat.sh with env: PAPERCLIP_AGENT_ID, PAPERCLIP_API_URL
→ Script calls openclaw agent --session-id main --message "[Paperclip Heartbeat] ..."
→ OpenClaw wakes the main session, processes the wake event
→ Agent replies: HEARTBEAT_OK
→ Script exits 0 → Paperclip marks run succeeded

Results — Engineering Notes
Blockers Hit → All Resolved
  • Embedded Postgres rejects root user→ Docker postgres:16-alpine workaround
  • missing scope: operator.write on heartbeat WS→ Root-caused: disableDeviceAuth:true strips all scopes. Fixed with stable Ed25519 keypair + device approval
  • invalid agent params: unexpected property 'paperclip'→ Schema mismatch: paperclipai 2026.416.0 sends fields OpenClaw 2026.3.28 rejects. Updated OpenClaw to 2026.4.15
  • openclaw_gateway adapter still fails after update→ Switched to process adapter + CLI bridge. Cleaner, no WS schema dependency
  • OpenClaw agent command needs session target→ --session-id main routes to the main active session
Known gap

The openclaw_gateway adapter needs openclaw to accept a paperclip field in agent params. Filed as a watch item — likely a minor schema update in a future OpenClaw release.
